一、能不能直接给别人用？

✅ 可以直接用（开发者模式加载）

代码完整，测试通过，无构建步骤。用户下载仓库 → chrome://extensions → 开发者模式 → Load unpacked → 即可运行。

⚠️ 上架 Chrome Web Store 需要处理几个小问题后才能提交

核心功能就绪，但有几处需要调整（见下方清单）。不涉及二次开发/重写，只是配置和文案层面的小修补。

上架待办清单

manifest.json description 改为中文 当前是英文 "Detect adult-solicitation tweets on X..."，但 popup 和 listing 都是中文。上架中文区建议统一，或用 __MSG_ 国际化。这是最显眼的不一致。
确认 Windows 上 zip 命令可用 npm run package 用的 zip -r。Git Bash 自带 zip，但原生 CMD/PowerShell 没有。可改用 PowerShell Compress-Archive 或手动压缩。也可直接在 chrome://extensions 点 "Pack extension" 生成 .crx。
注册 Chrome Web Store 开发者账号 一次性 $5 注册费。用 Google 账号登录 Developer Dashboard，提交 zip 包、填写 listing（文案已备好）、上传截图和 promo 图。
隐私政策 URL 可访问性确认 当前引用 GitHub blob URL。Chrome Web Store 要求一个可公开访问的 URL，GitHub 页面满足要求。但部分审核员偏好独立域名托管。如被拒，可考虑 GitHub Pages 部署一份。
建议添加 "minimum_chrome_version": "116" MV3 Service Worker 的某些 API（如 chrome.runtime.onMessage 异步 sendResponse）在 Chrome 99+ 才稳定。设 116 可覆盖绝大多数用户，避免旧版兼容问题。非必须但推荐。

二、功能完善建议

高优先级（影响用户体验和留存）

高 1. 被屏蔽账号列表管理 UI content.js → popup.html

blockedHandles 存了所有已屏蔽的 handle，但 popup 里没有入口查看或移除。用户误屏蔽后只能去 X 设置页手动找。建议在 popup 加一个折叠区域展示列表，支持单条删除（从 settings.blockedHandles 移除即可，X 端的 unblock 仍需用户手动操作或提示去 X 设置页）。

高 2. 设置导入 / 导出 popup.js

白名单、关注保护、阈值配置全存在 chrome.storage.local。重装扩展、换电脑、多 Chrome profile 都会丢失。加一个"导出 JSON"和"导入 JSON"按钮即可，数据结构本身就是 settings 对象。

高 3. 用户自定义关键词 detector.js → popup

当前检测规则完全靠内置正则 + 远程 keywords.txt。用户遇到新型引流话术时只能等维护者更新。建议在 popup 加一个"自定义屏蔽词"文本框，存储后作为 remoteKeywords 的补充传入 evaluateTweet。findRemoteKeyword 逻辑可直接复用，几乎零成本。

高 4. 误报反馈通道 content.js → banner

Banner 有"显示 / 屏蔽 / 信任"三个按钮，但没有"误报"反馈。用户觉得检测错误时没有直接通道告诉维护者。可以在"信任"旁边加一个"反馈误报"按钮，复制推文 handle + 检测原因到剪贴板并打开 GitHub Issue（复用社区贡献的 issue 流程，换一个 template）。

高 5. 扩展图标 Badge 计数 background.js

自动屏蔽在后台静默执行，用户浏览 X 时可能完全没注意到。建议用 chrome.action.setBadgeText 在扩展图标上显示本轮已屏蔽数（如 "3"），点击 popup 后清除。不需要额外权限，action 权限 MV3 自带。

中优先级（提升健壮性和覆盖面）

中 6. i18n 国际化框架 popup.html → manifest.json

Popup 全中文，但 x-ui-text.js 已支持中/英/日/韩四种语言的屏蔽菜单匹配。如果面向海外用户，popup 也应国际化。用 Chrome 原生 chrome.i18n + _locales/ 目录，或简单的 JS locale map。不急，但上国际市场前必须做。

中 7. 检测历史日志 content.js → popup

当前统计只有 4 个计数器（识别/排队/已屏蔽/失败）。加一个最近 50 条记录的环形缓冲区，存 { handle, score, category, time, action }。用户可以在 popup 里看到"最近屏蔽了谁、因为什么"，也方便排查误报。

中 8. 快速暂停 / 恢复 popup.js

用户可能临时想看完整时间线（比如做调研）。当前需要打开 popup → 取消勾选"启用检测" → 重新扫描。可以加一个显眼的"暂停 30 分钟"按钮，或右键菜单快捷暂停。

中 9. protected-accounts.txt 预填 blocklists/

当前为空。建议预填一批高关注度的官方/媒体/公众人物账号（如 @X, @TwitterSafety 等），防止社区报告误伤。也可以考虑自动把用户的 followedHandles 合并到保护逻辑（当前已有运行时保护，但远程账号列表的 100 分会覆盖保护——不过 evaluateTweet 开头已检查 whitelist/followedHandles 优先返回 protected，所以运行时没问题，只是远程列表聚合时需要 protected-accounts.txt）。

中 10. 远程列表完整性校验 remote-lists.js → background.js

从 GitHub Raw 下载 keywords.txt 和 accounts.txt 时没有 checksum 或签名验证。如果仓库被入侵或 GitHub 被劫持，攻击者可以注入任意 handle 到 accounts.txt，导致用户自动屏蔽合法账号。建议在仓库维护一个 SHA256SUMS 文件，下载后校验哈希。或者用 GitHub Releases 的 asset 附带签名。

低优先级（锦上添花）

低 11. 主题手动切换

当前用 prefers-color-scheme 自动跟随系统。部分用户想手动指定。加一个 light/dark/auto 三选一即可。

低 12. 首次使用引导

Popup 里的"隐藏分""屏蔽分""关注保护"等术语对非技术用户不友好。可以加一个 ? tooltip 或首次安装时弹出简短引导。

低 13. accounts.txt 添加元数据

当前只有 handle 列表，没有加入原因或日期。加注释行 # @handle 2026-07-22 community:3reporters 可以追溯。不过 parseAccountList 会过滤 # 开头的行，所以注释不会影响功能。

低 14. 社区贡献替代方案（非 GitHub 用户）

当前社区贡献需要 GitHub 账号。可以加一个"复制到剪贴板"的退路：用户复制 handle 列表后自行处理（发邮件、发 issue 等）。submitCommunityReports 已有剪贴板逻辑，只是之后强制打开 GitHub Issue 页面。可以加一个选项让用户选择是否打开 GitHub。

三、代码质量评价

维度评价说明
架构设计优秀7 个模块各司其职，通过 globalThis 挂载共享。无构建工具、无依赖、无 transpile，纯原生 JS，加载即跑。
检测引擎优秀纯函数 evaluateTweet 易测试，评分逻辑分层清晰（display name / body / reply / emoji / remote）。43 个测试覆盖正例和反例。
DOM 适配良好·脆弱依赖 X 的 data-testid 和 aria-label，X 改版会断。代码已有多语言 fallback 和宽泛选择器，但本质上无法避免。失败原因有文案提示，用户体验可接受。
隐私安全优秀不调私有 API、不收集推文正文、社区贡献默认关闭、无远程代码执行、无 GitHub token。权限最小化。
测试覆盖良好detector / remote-lists / community-sharing / mute-words / x-ui-text / community-aggregate 六个模块有测试。content.js 和 popup.js 无测试（需要浏览器环境，合理）。
错误处理良好核心流程有 try/catch，storage 读写有 fallback。但 boot() 如果抛异常会静默失败（只有 console.warn），用户看不到。建议加一个全局错误提示 banner。
性能良好MutationObserver + WeakSet 去重，只处理新增节点。stats 批量刷新（700ms debounce）。reEvaluateAll 在设置变更时触发全量重扫，但只在必要时调用。
架构亮点：整个扩展零依赖、零构建步骤。所有源码是可直接在浏览器运行的 IIFE 模块，通过 globalThis.Xyb* 命名空间共享。测试用 Node.js 内置 node --test，不需要 Jest/Mocha。这在 Chrome 扩展生态里是非常干净的实现。
最大风险：X DOM 结构变更。findMoreButton、findBlockMenuItem、findConfirmBlockButton 三个函数依赖 data-testid="caret"、data-testid="confirmationSheetConfirm" 等选择器。X 历史上会不定期改这些 testid。建议在 README 或 popup 里加一个"检测失效怎么办"的说明，并考虑加一个自动检测 selector 失效的通知机制。
